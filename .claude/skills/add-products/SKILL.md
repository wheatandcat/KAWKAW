# 定期商品追加ワークフロー

## トリガー

ユーザーが「定期商品の追加を作業をして」と言ったとき、このスキルを実行する。

## 概要

架空ECサイト「カウカウ」に毎週20件の商品を追加するワークフロー。

- **フェーズ1**: 生成・レビュー・plans/保存・Discord通知（自動）
- **フェーズ2**: 承認商品をコードに追加（ユーザー指定後に実行）

---

## フェーズ1: 商品生成・レビュー・保存

### Step 1: 事前情報収集

以下を読み込む:

1. `apps/public/lib/custom-prompt.ts` — 商品生成プロンプト (CUSTOM_PROMPT)
2. `apps/public/lib/products.ts` — 既存商品の `name` フィールドすべて（重複チェック用）
3. 既存商品の最大IDを確認し、新商品IDの開始番号を決める（最大ID + 1）

### Step 2: 商品20件を生成

CUSTOM_PROMPT の指示に従い、以下の制約を守って20件の架空商品を生成する:

**生成時の追加制約**:

- 既存商品名と実質的に同じコンセプトの商品は除外する
- 同じ「笑いのパターン」が5件以上連続しないようにする
- カテゴリを分散させる（テクノロジーに偏らない）
- `category` は以下から選択: テクノロジー / ファッション / 食品・グルメ / ホーム・生活 / ヘルス・ビューティー / エンタメ・趣味
- `image` は kebab-case の Lucide アイコン名で指定

### Step 3: Claude による自己レビュー（レビュアー1）

生成した20件について以下を評価する:

| チェック項目 | 基準                                                             |
| ------------ | ---------------------------------------------------------------- |
| 倫理チェック | 差別・暴力・実在ブランド侵害・核や兵器・人の命を奪うものがないか |
| 重複チェック | 既存商品または今回の20件内で実質的に同じコンセプトがないか       |
| ユーモア品質 | 笑いのパターンが多様で独自性があるか                             |
| 総合判定     | ✅承認 / ⚠️要修正（修正して再判定） / ❌却下（リストから除外）   |

❌却下・⚠️要修正があった場合は、代替商品を生成して差し替え、最終的に20件が揃うようにする。

### Step 4: Claude Code サブエージェントによるレビュー（レビュアー2）

Agent ツールを使って独立したサブエージェントを起動し、以下のプロンプトでレビューを依頼する:

> あなたは架空ECサイト「カウカウ（KawKaw）」の厳格なコンテンツ審査員です。
> 以下の商品候補リスト（20件）を独立した視点で審査してください。
>
> **審査基準**:
>
> 1. 倫理的問題（差別・暴力・不快なコンテンツ）がないか
> 2. ユーモアの独自性と品質（1〜5点）
> 3. リスト内での類似コンセプト重複
> 4. 各商品への判定: ✅承認 / ⚠️要修正 / ❌却下
> 5. 総評（全体バランス、改善提案）
>
> 商品リスト:
> {生成した20件の商品名と説明を貼り付ける}

サブエージェントのレビュー結果を受け取り、指摘事項があれば商品を修正・差し替えする。

### Step 5: Codex MCP によるレビュー（レビュアー3）

`mcp__codex__codex` ツールを使って以下のプロンプトでレビューを依頼する:

**プロンプト**:

> あなたは架空ECサイト「カウカウ（KawKaw）」のコンテンツ審査AIです。
> このサイトは買い物依存症防止を目的としたユーモラスな架空ECサイトです。
> 商品はすべて架空で、実際の販売はありません。
>
> 以下の商品候補リストを審査してください。各商品について：
>
> 1. **倫理チェック**: 差別・暴力・不快なコンテンツ・実在ブランドの侵害がないか
> 2. **ユーモア品質**: 独自性と笑いのクオリティ（1〜5点）
> 3. **重複リスク**: リスト内での類似コンセプトの重複
> 4. **総合判定**: ✅承認 / ⚠️要修正 / ❌却下
>
> 各商品に1〜3行でコメントし、最後に総評（全体のバランスや改善点）を述べてください。
> 厳格かつ公平に審査してください。
>
> ---
>
> {生成した20件の商品名と説明（TypeScriptブロックを除いたテキスト）を貼り付ける}

レビュー結果を `plans/YYYY-MM-DD/codex-review.md` に保存する:

```markdown
# Codex レビュー結果

{mcp**codex**codex の返答内容}
```

Codex の指摘事項を反映して商品を修正・差し替えする。

### Step 6: plans/ に最終商品リストを保存

3レビュアーの指摘を反映した最終20件を以下の形式で保存する:

**ファイルパス**: `plans/YYYY-MM-DD/products.md`（YYYY-MM-DD は本日の日付）

**ファイル形式**:

````markdown
# カウカウ 商品追加候補 - YYYY-MM-DD

## レビューサマリー

- Claudeレビュー: [承認XX件 / 修正YY件 / 却下ZZ件]
- Claude Agentレビュー: [主な指摘事項の要約]
- Codexレビュー: [主な指摘事項の要約]

---

## 商品001: {商品名}

**ステータス**: ✅ 承認

### 説明

{商品説明}

### データ

```typescript
{
  id: "XXX",
  name: "{商品名}",
  price: NNNNN,
  originalPrice: NNNNN,
  description: "{商品説明}",
  category: "{カテゴリ}",
  image: "{kebab-case-icon}",
  rating: N.N,
  reviewCount: NNNN,
  badge: "{バッジ}",  // badgeなしの場合はこの行を省略
}
```
````

### レビュー結果

| レビュアー     | 判定 | コメント |
| -------------- | ---- | -------- |
| Claude（自己） | ✅   | ...      |
| Claude Agent   | ✅   | ...      |
| Codex          | ✅   | ...      |

---

（商品002〜020 も同じ形式で続ける）

````

### Step 7: Discord に投稿

```bash
export $(grep -v '^#' apps/public/.env | xargs) 2>/dev/null
python3 scripts/notify-discord.py plans/YYYY-MM-DD/products.md
````

### Step 8: フェーズ1完了をユーザーに報告

以下を報告する:

- 生成・承認された商品数
- plans/YYYY-MM-DD/products.md のパス
- codex-review.md のパス
- Discord 投稿の完了
- 次のフェーズ2の案内: 「Discordと plans/YYYY-MM-DD/products.md を確認後、追加したい商品番号を教えてください（例: 商品001, 003, 007）」

---

## フェーズ2: 承認商品をコードに追加

ユーザーが追加したい商品番号（例: 001, 003, 007）を指定したら実行する。

### Step 1: 対象商品のデータを確認

`plans/YYYY-MM-DD/products.md` から指定された商品のTypeScriptデータを読み込む。

### Step 2: `apps/public/lib/products.ts` を更新

以下の2箇所を更新する:

**① `productIcons` オブジェクトに追加**（末尾の `}` の直前に追記）:

```typescript
"XXX": "PascalCaseIconName",
```

※ image フィールドの kebab-case を PascalCase に変換する（例: `wand-2` → `Wand2`）

**② `products` 配列の末尾に追加**:

```typescript
{
  id: "XXX",
  name: "...",
  ...
},
```

### Step 3: 商品アイコンに漏れがないか確認

- apps/public/server/lucide-svg-data.tsにアイコンデータが無いものがある確認してください

### Step 4: TypeScript 型チェック

```bash
cd apps/public && npm run check
```

エラーがあれば修正する。

### Step 5: 完了報告

追加した商品名・件数・次のステップ（git commit → PR作成）を報告する。

---

## 注意事項

- 環境変数は毎回 `apps/public/.env` から読み込む
- plans/ ディレクトリは git 管理する（商品追加の履歴として保存）
- 商品IDは連番で、既存の最大IDの次から振る
- image フィールドは kebab-case、productIcons は PascalCase（Lucide の命名規則）
